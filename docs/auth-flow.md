# Authentication Flow and User ID Management

This document explains how the authentication flow works and how user IDs are managed in the translation app.

## Key Components

### 1. Google Provider Account ID
- **What it is**: Google's stable user identifier (e.g., `108123456789`)
- **Where it comes from**: Google OAuth response during sign-in
- **Lifetime**: Permanent - never changes for a user
- **Stored in**: `app_user.provider_user_id` and `JWT token.providerAccountId`

### 2. Internal User UUID
- **What it is**: Our database's internal user identifier (e.g., `abc123-def456-...`)
- **Where it comes from**: Generated by PostgreSQL on first user creation
- **Purpose**: Primary key for all user-related data
- **Stored in**: `app_user.id` and `session.user.id`

### 3. JWT Token
- **What it contains**: `{ providerAccountId: "108123456789", sub: "...", ... }`
- **Purpose**: Carries Google's identifier in the encrypted session token
- **Lifetime**: Session duration (managed by NextAuth)
- **Why we need it**: Enables lookup of internal UUID on every request

### 4. Session Object
- **What it contains**: `{ user: { id: "UUID", email: "...", ... } }`
- **Where it comes from**: Session callback queries DB using providerAccountId
- **Purpose**: Provides internal UUID to all API routes
- **Access**: `await auth()` or `useSession()` hook

## Authorization Flow

### Every API Request:
1. Browser sends session cookie automatically
2. API route calls `await auth()`
3. NextAuth:
   - Decrypts JWT token
   - Extracts `token.providerAccountId`
   - Queries database: `getUserIdByProviderUserId("google", providerAccountId)`
   - Returns session with internal UUID
4. API route uses `session.user.id` (UUID) for all database operations
5. Database queries are scoped: `WHERE user_id = session.user.id`

### Data Isolation:
- All user data is scoped by internal UUID
- No user can access another user's translations
- App-layer authorization (no Row Level Security needed)

## Critical Callbacks in src/auth.ts

### signIn Callback
```typescript
async signIn({ user, account }) {
  // Creates or updates app_user record
  await upsertUser({
    provider: "google",
    providerUserId: account.providerAccountId,
    email: user.email,
    name: user.name,
    imageUrl: user.image
  });
  return true;
}
```

### JWT Callback
```typescript
async jwt({ token, account }) {
  // Stores Google's ID in JWT token
  if (account?.providerAccountId) {
    token.providerAccountId = account.providerAccountId;
  }
  return token;
}
```

### Session Callback
```typescript
async session({ session, token }) {
  // Looks up internal UUID and attaches to session
  const userId = await getUserIdByProviderUserId(
    "google",
    token.providerAccountId
  );
  if (userId) {
    session.user.id = userId; // Internal UUID
  }
  return session;
}
```

## Database Schema

### app_user Table
```sql
CREATE TABLE app_user (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  provider TEXT NOT NULL,
  provider_user_id TEXT NOT NULL,  -- Google's providerAccountId
  email TEXT,
  name TEXT,
  image_url TEXT,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  last_login_at TIMESTAMPTZ,
  UNIQUE(provider, provider_user_id)
);
```

### translation_history Table
```sql
CREATE TABLE translation_history (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES app_user(id) ON DELETE CASCADE,
  source_lang TEXT NOT NULL,
  target_lang TEXT NOT NULL,
  input_text TEXT NOT NULL,
  response_json JSONB NOT NULL,
  model TEXT,
  prompt_version TEXT,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX translation_history_user_created_at_idx
  ON translation_history(user_id, created_at);
```

## Troubleshooting

### Getting 401 Unauthorized Errors?

1. **Check if user exists in database:**
   ```sql
   SELECT * FROM app_user WHERE provider = 'google';
   ```

2. **Sign out and sign back in** - This ensures you get a fresh JWT token with `providerAccountId`

3. **Check server logs for:**
   - `[JWT] Stored providerAccountId in token: <id>` - Confirms JWT callback ran
   - `[Session] Token providerAccountId: <id>` - Confirms token has the ID
   - `[Session] Attached userId to session: <uuid>` - Confirms lookup succeeded
   - `[Session] Failed to find user in DB` - User record doesn't exist
   - `[Session] Missing providerAccountId in token` - Need to re-authenticate

4. **Enable debug mode** - Set in `src/auth.ts`:
   ```typescript
   debug: process.env.NODE_ENV === "development"
   ```

### User Not Found in Database?

Check if `signIn` callback is executing:
- Look for "Failed to upsert user" errors in server logs
- Verify database connection is working
- Check that `account.providerAccountId` is present in signIn callback

### Session Has No User ID?

- Verify JWT callback is storing `providerAccountId`
- Check that session callback is looking it up
- Ensure database query in `getUserIdByProviderUserId` is correct
- Verify `app_user` table has matching records

## Summary

The authentication system uses a two-stage ID mapping:
1. **Google → Database**: On sign-in, maps Google's `providerAccountId` to internal `UUID` in `app_user` table
2. **JWT → Session**: On each request, JWT token's `providerAccountId` looks up internal `UUID` and attaches to session

All API routes use the internal `UUID` from `session.user.id` to scope database queries, ensuring complete data isolation between users.
